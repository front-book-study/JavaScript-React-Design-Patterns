### 문제의 유형에 따라 분류된 디자인패턴들

#### 생성패턴

| 생성패턴                  | 객체 생성의 복잡도 문제 해결에 중점              |
|-----------------------|-----------------------------------|
| 팩토리메서드 (클래스)          | 인터페이스 기반으로 파생 클래스 생성              |
| Abstract Factory (객체) | 구체적인 내부구현 X, 여러 클래스가 상속받는 인스턴스 생성 |
| Builder (객체)          | 객체를 생성하는 부분과 내부 구현을 분리해 같은 객체를 생성 |
| Prototype  (객체)       | 복사/복제에 사용되는 초기화된 인스턴스             |
| Singleton  (객체)       | 전역에서 접근 가능한 하나만의 인스턴스를 가진 클래스     |


#### 구조패턴

| 생성패턴            | 객체 간 구성과 관계에 인식하는 방법에 중점          |
  |-----------------|-----------------------------------|
| Adapter (클래스)   | 호환되지 않는 인터페이스가 상호작용하도록 클래스를 매치    |
| Bridge (객체)     | 객체의 이넡페이스와 구현을 분리하여 독립으로 구성       |
| Composite (객체)  | 단순히 합친 상태 이상의 효율을 내는 간단하면도 복잡한 구조 |
| Decorator  (객체) | 객체에 새로운 프로세스를 동적으로추가              |
| Facade  (객체)    | 전체 시스템의 복잡한 부분을 숨기는 단일클래스         |
| Flyweight(객체)   | 여러 객체에 공통 상태를 공유하는 세분화된 인스턴스      |
| Proxy (객체)      | 실제 객체를 대신하는 대체 객체                 |


#### 행위패턴

| 생성패턴                         | 객체 간 커뮤니케이션을 개선하거나 간소화에 중점                  |
  |------------------------------|---------------------------------------------|
| Interpreter (클래스)            | 언어의 목적과 문법에 일치하는 언어 요소를 포함시키는 방법            |
| Template method (클래스)        | 상위 클래스에서 기본 구조를 생성한 다음 하위 클래스에서 구체적으로 정의    |
| Chain of responsibility (객체) | 요청을 처리할 수 있는 객체를 찾기 위해 체인 간에 요청을 전달         |
| Iterator  (객체)               | 내부구조를 모른 채 요소에 순차적으로 접근                     |
| Mediator  (객체)               | 클래스가 서로를 직접적으로 참조하지 않도록 중간에 간소화된 커뮤니케이션을 정의 |
| Memento(객체)                  | 나중에 복구할 수 있도록 객체의 내부 상태를 저장                 |
| Observer (객체)                | 클래스간의 일관성을 보장하기 위해 여러 클래스에 변경사항을 알리는 방법     |
| Strategy(객체)                 | 클래스 내부에 알고리즘 구현을 캡슐화하여 사오하엥 따른 선택과 구현을 분리   |
| Visitor (객체)                 | 클래스를 변경하지 않고도 새로운 작업을 추가                    |
