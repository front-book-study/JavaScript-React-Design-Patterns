### MV* 패턴

- 모델 : 비즈니스 로직, 데이터, 검증 로직 담당 (ex. state, validation etc ..)
- 뷰 : 화면 자체 (ex. html, css ..)
- 모델과 뷰 사이에서 중재자 역할을 하는 다양한 방식이 등장.
    - controller → mvc
    - presenter → mvp
    - viewmodel → mvvm

### **MVC (Model–View–Controller)**

**핵심 아이디어**

- Controller가 사용자의 입력을 받아서 Model을 변경하고, View는 Model을 읽어서 화면을 그린다.

**흐름**

- 사용자 입력 → **Controller** → **Model** 업데이트 → View가 Model을 기반으로 다시 그림

**특징**

- Controller는 “요청 처리 + 어떤 View를 보여줄지” 를 결정
- View는 “표현(렌더링)”에 가깝고 로직은 거의 없음
- Model은 **비즈니스 로직 + 상태**를 캡슐화

**장점**

- 역할 분리가 명확해 “서버에서 페이지 렌더링”할 때 이해가 쉽다.
- Controller 단위로 테스트하기 괜찮다.
- 라우팅(요청 → 컨트롤러 매핑) 구조와 잘 맞는다.

**단점**

- View ↔ Controller 간 결합이 높아지기 쉽다.
- 큰 프론트엔드 앱에서는 **Controller가 비대해지기** 쉽다.
- 양방향 바인딩이 없어서, 상태 변경 후 수동으로 View를 갱신해줘야 하는 경우가 많음.

### **MVP (Model–View–Presenter)**

**핵심 아이디어**

- View는 “인터페이스”만 제공,
- Presenter가 **모든 입력 처리/비즈니스 로직/Model 조작/화면 업데이트**를 담당한다.

**흐름**

- 사용자 입력 → **View**가 이벤트를 Presenter에게 위임 → Presenter가 Model 조작 → Presenter가 **View 인터페이스를 통해 화면 업데이트 호출**

**특징**

- View는 “버튼 눌림 전달”과 “Presenter가 명령한 대로 그리기만” 한다.
- Presenter에 UI 로직이 집중 → **테스트가 쉬움** (View를 mock으로 대체 가능)
- Model은 MVC와 비슷하게 도메인 로직 담당.

**장점**

- View를 인터페이스로 추상화해서 **Presenter 단위 단위 테스트**가 쉬움.
- View 로직과 화면을 교체하기 좋다 (웹→모바일 같은 경우).
- Android에서 Activity/Fragment(View)와 Presenter 분리가 흔한 패턴.

**단점**

- Presenter가 비대해지기 쉬움 (Fat Presenter).
- 데이터 바인딩이 없어서 View 업데이트를 전부 Presenter가 직접 호출해야 한다.

### **MVVM (Model–View–ViewModel)**

**핵심 아이디어**

- ViewModel은 **상태와 액션**만 제공하고,
- View는 데이터 바인딩으로 ViewModel을 “자동으로 반영”한다.

**흐름**

- 사용자 입력 → View에서 ViewModel의 메서드/명령 호출 → ViewModel이 Model을 변경 → View는 ViewModel의 observable state를 **자동으로 반영**

**특징**

- ViewModel은 View를 알지 못하고, **오직 데이터와 이벤트만 노출**.
- View는 데이터 바인딩(템플릿 언어, 컴파일러, 프레임워크)이 관여.
- React의 state + 렌더링, Vue의 reactive state, Angular의 template 바인딩 등이 이 감각.

**장점**

- **상태 중심**으로 사고 → 복잡한 화면 상태를 정리하기 좋음.
- ViewModel 테스트가 쉽다 (View 없이 상태·메서드만 테스트).
- UI 업데이트가 “상태 변경”에 의해 자동으로 일어나서 코드 노이즈 감소.

**단점**

- 바인딩 시스템이 복잡하면 디버깅이 어려울 수 있다.
- 패턴이 추상적이라, 팀 합의 없이 쓰다 보면 구조가 애매해지기 쉽다.
- 단순한 화면에는 오히려 과한 구조일 수 있다.
